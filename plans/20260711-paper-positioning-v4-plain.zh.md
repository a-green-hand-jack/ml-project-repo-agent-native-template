## 一句话：我们做了什么

我们做了**一套"给 AI agent 干活的地方定规矩"的框架**。

不是给 agent 写一段"你要乖"的提示词，也不是造一条固定的流水线让它照着走。而是：**把规矩、边界、证据要求、状态记录，直接刻进项目仓库本身，让它们能被脚本自动检查。** agent 一进这个仓库，就被环境本身约束着——哪些东西不能碰、结论必须有证据、干到一半换个人/换个 AI 也能接上、结构不能悄悄烂掉。

一句口号：**"设计干活的环境，而不是指望 agent 自觉。"**（Design the harness, not the agent.）

---

## 为什么需要它：三个真实的痛

让 AI 在一个**长期、高风险、要出真东西**的项目里干活，会反复踩三个坑：

1. **它会忘。** 上下文一长、一压缩，早先说好的目标和边界就糊了。规矩只写在聊天里，下一轮就成了传闻。
2. **它会飘、会闯祸。** 它可能顺手删掉训练了三天的 checkpoint、把回测结论说得比数据支持的更满、改了目录却不更新说明、悄悄把"草稿结果"写成"论文结论"。
3. **光靠"提示词让它别这样"不管用。** 研究表明，AI 对提示词的遵守程度会随着任务变长而下降——恰恰在最需要它守规矩的长任务里最不守。

所以我们的判断是：**真正要它必然做到的事，不能靠嘴说，得靠环境卡住。** 想让它理解偏好，用文档；想让某件事必然发生或必然被拦，用仓库里的脚本和权限。

## 核心思路：管"地方"，不管"你怎么干活"

这是这版最重要的一点，也是之前几版没说对的地方。

**我们管的是"干活的地方"（工作空间 / lab），不是"你怎么干活"（工作流程）。**

打个比方：我们提供的是**一间实验室的规章 + 安全护栏 + 台账**，而不是一条"必须先做A再做B再做C"的装配线。在这间实验室里，你可以按部就班，也可以东试试西试试、推倒重来、并行开好几条线、临时造个工具——**规矩不要求你排成一条线**。护栏、证据要求、交接记录、防乱改，是**裹在**你灵活干活的外面，而不是把你**压成**一条流水线。

用一句话说清楚我们的身份：
> **它可以"罩住"一条流水线，但它不是流水线生成器；它可以承载一套工作流，但它不是工作流产品。**

而且——**产出是开放的，上限是"一个 agent 本身"。** 你在这间实验室里可以产出一个指标、一个因子、一个数据集、一个模型、一个科学发现，**也可以产出一个工具、一套评测框架，甚至一个完整的 agent。** 框架只管"产出是怎么形成的、有没有证据撑、交接清不清楚、越界动作有没有拦"，**不规定你要产出什么**。

## 能用在哪：四个场景，一个道理

这套东西不是只给机器学习用。凡是**"高风险、周期长、多人/多 agent、要靠证据说话、产物一堆、有些动作干了就收不回"**的地方，都能用。我们打算在四类上验证：

| | 在这个领域里，"实验"是 | "结论"是 | "收不回的动作"是 |
|---|---|---|---|
| **机器学习研究** | 训练、评测、调参 | 模型好不好、机制解释 | 重启长训练、覆盖 checkpoint |
| **量化研究** | 回测、因子检验、建模 | 挖到的算子/因子/一个好模型、alpha 稳不稳 | **实盘下单**（收不回的是真钱） |
| **计算科学** | 仿真、HPC 作业、扫参 | 一个发现、一个数值结论 | 提交/继续烧一个巨贵的算力作业 |
| **造 agent / 工具** | 收集轨迹、跑 benchmark、红队 | 这个 agent 能力多强、够不够安全、能不能发布 | 部署一个生产/交易 agent |

**同一套规矩，套四种不同的"业务含义"。** 这就是它的通用性——不是把四个领域的差异抹平，而是证明**同一个骨架能被每个领域"填进自己的细节"**。

关于量化多说一句（因为容易被人抬杠）：我们管的是**量化研究**（回测、挖因子、建模），不是替代实盘系统。实盘下单这种"收不回"的动作，正是我们"边界"这一块要盯的地方——它得跟券商风控、IAM 这些"更硬的外部关卡"配合，仓库里的护栏**不假装自己能保护真金白银**。

关于"造 agent"也多说一句：**这个模板自己就是活例子**——我们正是用这套框架，来管"造一个 agent 开发工具箱"这件事。所以"用框架造 agent"不是空话，是自证。（诚实讲：这只是"自己用自己"，能说明好用、能演化，但不能单独证明它对别人也有效。）

## 方法长什么样：管 5 样东西，一直做 2 件事

别人要能照着用，方法就得能脱离我们这个仓库讲清楚。我们把它收成——**5 样"必须在环境里管住的东西" + 2 件"一直在后台做的事"。**

**要管住的 5 样：**

1. **边界**：谁能对什么资产做什么动作，哪些动作昂贵/不可逆要特别标记、要人来批。（比如"不许删 lab/data"、"推 main 要放行"。）
2. **证据**：每个结论都得挂上它的来源（哪个 run、哪个 config、哪个指标），**结论不能说得比证据更满**，说满了就报错。
3. **结构地图**：目录怎么组织、谁负责哪块、哪能写哪不能写——**改了结构就得同步改地图**，对不上就是"漂移"，报错。
4. **接续状态**：当前目标、决定、没解决的风险、下一步，都落在文件里，**换个 AI 从零上下文也能接着干**。
5. **能力包**：项目自带的 agent/技能/脚本/检查器，一套源头，能自动同步到不同 AI 平台（Claude、Codex），两边规矩一致。

**一直在做的 2 件事：**

- **持续自检**：脚本不断检查上面 5 样有没有破（引用断没断、脚本加载没加载、故意塞个违规能不能被拦住）。
- **受管演化**：规矩本身也会改；改的时候要带上原因、复核、迁移、复测——让"规矩变了"这件事是显式、可控的，而不是偷偷变。

一句话把它和"形状无关"接上：**这 5 样只要求"这些东西在环境里能被认出来、能被检查"，不要求你的活按 1→2→3→4→5 的顺序走。** 你可以先造个评测工具再回头改假设，可以废掉一条量化分支，可以红队几轮后重写 agent 的行为规矩——只要没把边界、证据、归属、接续弄丢，框架就不把"你没按直线走"当成错误。

## 怎么在一个新项目里用起来：五步

（注意：这五步是**给治理框架做配置校准**的步骤，不是规定你的研究要按这五步走。）

1. **摸清楚这个项目有哪些"会出事的地方"**：画出会反复出现的活动和风险（误启动、误删、越权把草稿变结论、失联、乱改……），每个风险标上"多严重、能不能撤、谁负责"。
2. **把风险翻译成可检查的规矩**：哪条归到上面 5 样里的哪一样，定出"证据要到什么程度算够"、"哪些是通用规矩、哪些是这个领域特有的检查"。
3. **决定每条规矩"在哪把关、怎么恢复"**：是提醒、是直接拦、是事后检测、还是支持出事后回滚；越贵越不可逆的动作，越往"更外层更可信的关卡"放。
4. **把能力编译到具体平台**：生成 Claude/Codex 各自的脚本和适配；编译不了的能力要**明说降级**，不许假装通用。
5. **跑一遍检验 + 定期复测**：正常路径、故意违规、故意不加载脚本、故意丢上下文、故意改规矩——都试一遍，看拦没拦住、能不能恢复；记录下漏报误报和维护成本。

判定"配好了"的标准：每个高风险点都至少有一个关卡 + 一条恢复路，核心检验能过，**至少有一种"换个干法"不会把治理弄坏**，做不到的地方老实记下来而不是藏起来。

## 和别人的工作什么关系：不是打架，是搭积木

有一篇很近的工作（*A Deterministic Control Plane for LLM Coding Agents*）——**先纠正我们自己之前的错**：它其实做得很扎实，有上万个仓库的实证、有故意注违规的一致性测试、能从一套定义编译到七个平台、还有签名和防篡改的审计日志。**我们之前说它"没实证、单平台"是错的，撤回。**

那我们跟它到底差在哪？**差在"管的东西不一样"，而且我们能踩在它肩膀上：**

- 它管的是**通用编程 agent 的编码活动**（权限、阶段、agent 定义的供应链完整性）。
- 我们管的是**研究/构建实验室里的证据、产物、接续、和那些收不回的动作边界**——而且横跨 ML/量化/科学/造 agent。
- 关系是**搭积木，不是抢地盘**：它那套完整性机制我们可以直接拿来当底座，我们在上面补"研究证据语义 + 跨 session 接续 + 灵活产出的生命周期"。最强的对照实验甚至是"它的配置 + 我们的研究这层"。

所以我们的卖点**不是"我们比它多做了实验"**（实验谁都能补，那不是护城河）。卖点是：**在一个可靠的通用控制面之上，一个"实验室"还额外需要哪些东西，而且这些东西能不能跨研究领域、甚至跨到"造 agent"复用。**

还有一篇（Russo，《Govern the Repository, Not the Agent》）——标题跟我们很像，但它是**测量**"该在仓库层面治理"这件事的证据，**没造出那个东西**。它是我们最好的"问题背书"，不是竞品。

## 怎么证明它有用：诚实的实验计划

现在的真实状态：**ML 有实现和一些回放证据；"造 agent"有本模板自身这个纵向例子；量化和计算科学还是 0 个真实实例。** 所以现在**只能说"框架定义 + 方法 + 可行性计划"，不能说"已经证明普遍有效"。**

打算怎么验：四个领域**跑同一套测试**，只换各自的领域细节——

1. 从风险配出那 5 样规矩，记录花了多久、哪里含糊；
2. 在两个 AI 平台上跑同样的任务，看行为一不一致；
3. **故意塞违规**（写受保护文件、越权、拿没证据的结论去升级……），看拦没拦住；
4. **故意丢掉上下文**，让一个"新手 AI"只靠文件恢复，看能不能接上；
5. **故意改规矩/结构**，看能不能检测、迁移、回滚；
6. 从产出造一个候选结论，验证证据够不够、要不要人批；
7. **故意换个干法**（中途分支、回退、改产出类型），看治理还成不成立、有没有把合法的灵活干法误当违规。

对照组：只用提示词/AGENTS.md、用那篇竞品的等价配置、用完整框架。还要**至少找一个外部团队"盲着"照论文配一遍**，证明这不是"只有作者自己会用"。

## 诚实地说它做不到什么

这块很重要，别被审稿人抓：

- 仓库里的护栏是**防手滑的栏杆，不是防黑客的沙箱**；`--no-verify` 这类逃生口、受信任的代码、直接调外部系统，都可能绕过去。我们**只承诺在说清楚的假设下、对已经写进去的规矩**提供保障，不承诺"绝对不能违反"。
- 真正涉及钱、云、生产部署的硬边界，得靠券商风控、IAM、部署系统这些**外部更硬的关卡**——仓库不冒充它们。
- "形状无关"也不是白拿的：任何 schema 都有填写成本，任何关卡都可能拖慢探索。我们只说"不强迫你按固定流程走"，**不说治理对你的干活完全没影响**。
- 检查器**自己也可能被改坏**（自己验自己有递归信任问题），所以强的完整性得靠外部 CI / 签名兜底。

## 能发什么、投哪、叫什么

**这是一篇"以制品为中心的系统/方法论文"**——不是工作流产品、不是纯讲道理的立场文、也不是"目录堆得多"的模板文。

**节奏**（跟你定的一致）：先投**非归档的 workshop**（把框架、方法、评测计划、加一个 ML 小试验讲清楚，量化/科学明说"待做"），拿一波审稿意见；再补齐两个真异质领域 + 外部盲测 + 直接对照，投正会（CAIN / FSE / ICSE 这类软件工程/agentic 方向，比泛 agents 会更合身）。

**标题首选**：
> **Governing the Lab, Not Shaping the Work**（治理 Lab，而非规定工作形状）
> ——面向高风险、长时程 agentic 工作空间的治理框架。

它一句话说清了这次的转向：**我们提供治理框架，不把开放的活儿压成机器或流水线。**

## 还没想清楚的地方（留给实验和后续）

1. **"形状无关"和"总得有点结构"怎么平衡**——哪些是最小必要结构，哪些只是可选项？
2. **灵活 vs 保障**——护栏太松没用，太紧误伤探索，得用"误报率、重规划成功率、多出来的人工审批"来量，而不是嘴上说"不限制"。
3. **通用核心 vs 领域专用检查**的分界——不然"通用方法"退化成"每个领域各写一套"。
4. **造 agent 时，外层开发权限别泄漏到内层产品**——两层都有 policy/工具/评测，得有稳定的隔离。
5. **仓库护栏和外部硬关卡的分工**——越危险越该外部管，那仓库这层的"可测增量"到底是什么。
6. **通用性的证据门槛**——现在只够说"可行"，要说"普遍有效"得等外部盲测和跨域数据。

---

## 一句话总结

**我们不是造一台机器，也不是给 AI 写一段"要乖"的提示词。我们提供一套"实验室治理框架"：把边界、证据、接续、防乱改刻进仓库、让脚本自动守着，罩住 AI 灵活开放的干活方式——小到出一个指标，大到造出一个 agent——而不把这活儿压成任何固定流水线。**
